### Finding A-1: trim_null_effects.mf_individual docstring is now stale and the no-op may double-count with susie_get_pip's prior_tol filter
- **Severity**: track-for-later
- **Pathway**: effective V consumers
- **Location**: mfsusieR/R/ibss_methods.R:228-240; mfsusieR/R/individual_data_methods.R:983-1003 (cleanup_extra_fields docstring); susieR/R/model_methods.R:339-352 (trim_null_effects.default); susieR/R/susie_get_functions.R:486-535 (susie_get_pip)
- **Verified by reading**: yes — read both files end to end; cross-checked the V[l] write path through `set_prior_variance_l(model, l, V)` in `susieR/R/single_effect_regression.R:54`.
- **Description**: susieR's contract for `trim_null_effects.default` is to zero out alpha / mu / mu2 / lbf / KL for effects whose V[l] is below `params$prior_tol`, separately from `susie_get_pip`'s own `V > prior_tol` filter (used only for PIP construction). Before commit 605f395 the mfsusieR override was a documented no-op because V[l] was always 1; with V[l] now carrying the per-effect effective slab variance (mean over (m, s) of `sum_k pi[l, m, s, k] * var_k`), a near-null effect with `pi_null ≈ 1` and slab vars ~0.1 will hit V[l] ~ 1e-11, well below `prior_tol = 1e-9` (mfsusie.R:358). The mfsusieR override stays a no-op so alpha / mu / mu2 / lbf / KL are NOT zeroed for these effects, but `susie_get_pip` (called inside both `ibss_finalize` and per-iter `check_alpha_pip_cycle_convergence`) DOES drop them from the PIP. The docstring at ibss_methods.R:230-234 still claims "mfsusieR holds `V[l] = 1` for all l (mixture prior absorbs the per-effect adaptation), so the V-based pruning does not apply" which is no longer true.
- **Triage hint**: track-for-later
- **Suggested action**: Either (a) refresh the trim_null_effects.mf_individual docstring to acknowledge the new V[l] semantics and explicitly document that `susie_get_pip`'s filter is the only V-based pruner active, OR (b) make trim_null_effects.mf_individual perform the same alpha / mu / mu2 / lbf / KL zeroing for V[l] < prior_tol that the .default does (so the stripped fields are coherent with the dropped PIPs).

### Finding A-2: ibss_initialize.mf_individual skips susieR's validate_init for warm-start fits
- **Severity**: track-for-later
- **Pathway**: per-effect state
- **Location**: mfsusieR/R/ibss_methods.R:161-201; susieR/R/iterative_bayesian_stepwise_selection.R:54-77 (.default warm-start path); susieR/R/susie_utils.R:339-454 (validate_init)
- **Verified by reading**: yes — confirmed both paths line by line.
- **Description**: susieR's `ibss_initialize.default` calls `validate_init(data, params)` (iterative_bayesian_stepwise_selection.R:56) when a warm-start `model_init` is supplied; the validator checks that alpha / mu / mu2 / V / sigma2 / pi contain no NA/Inf and that dim(mu) == dim(alpha). mfsusieR's override (lines 161-201) does NOT call validate_init — only `expand_model_init_to_L`, which only checks that `mi$alpha` exists and `nrow(mi$alpha) <= L_new`. The omission is structurally necessary because mfsusieR's mu / mu2 are list-of-list rather than the matrix shape `validate_init` assumes (susie_utils.R:407 `if (!all(dim(mu) == dim(alpha)))` would crash on mfsusieR's mu shape). Net effect: warm-start fits supplied with NA/Inf in alpha or with mismatched mu list dimensions will only fail later, deeper in the IBSS loop, with a less informative error.
- **Triage hint**: track-for-later
- **Suggested action**: Add an mfsusieR-flavored validator alongside `expand_model_init_to_L` that mirrors validate_init's NA/Inf checks plus the list-of-list shape checks for mu / mu2 / pi_V / fitted_g_per_effect. Not urgent for correctness; useful for clearer failure modes.

### Finding A-3: post hook fetches optimize_prior_variance and inner-loop helpers via getFromNamespace, not via S3 dispatch
- **Severity**: needs-trace
- **Pathway**: hook contract
- **Location**: mfsusieR/R/individual_data_methods.R:701-714 (post_loglik_prior_hook.mf_individual inner loop); susieR/R/single_effect_regression.R:90-108 (.default post hook); susieR/R/model_methods.R:31-47 (get_alpha_l / get_posterior_moments_l)
- **Verified by reading**: yes — read mfsusieR's inner EM loop, susieR's .default hook, and the cached generic accessors.
- **Description**: susieR's `post_loglik_prior_hook.default` (single_effect_regression.R:97-108) calls `optimize_prior_variance(...)` via dispatch and forwards the result back. mfsusieR's `post_loglik_prior_hook.mf_individual` calls `optimize_prior_variance.mf_individual(...)` directly by name (line 711), bypassing S3 dispatch. It also fetches `loglik`, `calculate_posterior_moments`, `compute_kl`, `get_alpha_l`, `get_posterior_moments_l` via `getFromNamespace("...", "susieR")` (lines 703-707). The fetched generics still dispatch on data class because `UseMethod` consults the methods table at call time, so functionally this works. But (a) the `moments` argument that the post-hook passes (`get_post(model, l)`, line 714) is in mfsusie's `get_posterior_moments_l.mfsusie` shape `list(mu=, mu2=)`, not susieR's default `list(post_mean=, post_mean2=)` shape — `optimize_prior_variance.mf_individual` ignores the argument so this is currently dead-code; (b) any future subclass of `mf_individual` that wants to override `optimize_prior_variance` will be silently bypassed because of the direct `.mf_individual` call. Both are latent maintenance risks rather than current bugs.
- **Triage hint**: needs-trace
- **Suggested action**: Replace `optimize_prior_variance.mf_individual(...)` with the dispatched form `optimize_prior_variance(data, params, model, ser_stats, l = l, alpha = ..., moments = ..., V_init = V_init)`. Drop the unused `moments = get_post(model, l)` argument since `optimize_prior_variance.mf_individual` does not consume it.

### Finding A-4: Per-effect persistent state warm-load is consistent with the L_greedy expansion path
- **Severity**: accept-pre-existing
- **Pathway**: per-effect state
- **Location**: mfsusieR/R/ibss_methods.R:83-143 (expand_model_init_to_L); mfsusieR/R/ibss_methods.R:161-201 (ibss_initialize.mf_individual); mfsusieR/R/individual_data_methods.R:677-689 (pre_loglik_prior_hook.mf_individual); mfsusieR/R/individual_data_methods.R:761-829 (.opv_mixsqp); mfsusieR/R/individual_data_methods.R:844-868 (.opv_ebnm_point); susieR/R/susie_workhorse.R:21-62 (L_greedy outer loop)
- **Verified by reading**: yes — traced fitted_g_per_effect[[l]] from initialization (model_class.R:73-79) through pre-hook copy-in (lines 679-687) to mixsqp / ebnm write-back (lines 825-826, 864) to L_greedy round handoff (susie_workhorse.R:37 `params_round$model_init <- fit`) and warm-load (ibss_methods.R:172-184).
- **Description**: susieR's L_greedy contract supplies the round-`k` fit as `params_round$model_init` for round `k+1`, requiring the downstream package's `ibss_initialize` to honor pi_V / fitted_g_per_effect across rounds and append zero-state slots when L grows. mfsusieR threads this correctly: `expand_model_init_to_L` rebinds `mi$pi_V` and `mi$fitted_g_per_effect` to length `L_new`, seeding new effect slots with effect 1's prior state (lines 134-141); `ibss_initialize.mf_individual` then copies the warm fields into `mat_init` (lines 175-181). At the start of each effect's SER step, `pre_loglik_prior_hook.mf_individual` copies `fitted_g_per_effect[[l]]` back into the shared `G_prior` scratchpad (lines 679-687) so each effect sees its own persistent prior, and the M-step writes back into both `G_prior` AND `fitted_g_per_effect[[l]]` (lines 824-826, 864). No divergence; this satisfies the susieR / mvsusieR S3 paradigm.
- **Triage hint**: accept-pre-existing — pathway is clean.

### Finding A-5: Hook contract and S3 dispatch sanity confirmed; one minor coupling note
- **Severity**: accept-pre-existing
- **Pathway**: hook contract / dispatch sanity
- **Location**: mfsusieR/R/zzz.R:25-107; mfsusieR/R/individual_data_methods.R:677-726; susieR/R/single_effect_regression.R:17-108; susieR/R/iterative_bayesian_stepwise_selection.R:84-97
- **Verified by reading**: yes — read .onLoad's S3 method registration table, both hooks' return shapes (`list(V = ..., model = ...)`), and the SER orchestrator's consumption of those returns.
- **Description**: susieR's contract for both hooks is a `list(V = ..., model = ...)` return, with V being a numeric scalar that is then written to `model$V[l]` via `set_prior_variance_l(model, l, V)`. mfsusieR's pre-hook returns `list(V = 1, model = model)` (line 688), and the post-hook returns `list(V = .effective_V_l(model, l, data$M), model = model)` (line 725) where `.effective_V_l` returns a scalar. Both shapes match; no susieR consumer reads V semantically except (a) `format_V_summary(model$V)` in verbose output (model_methods.R:172, mfsusieR overrides format_extra_diag.mfsusie but the V column itself is fine), (b) `trim_null_effects.default` (mfsusieR overrides to a no-op — see Finding A-1 for the staleness issue), (c) `susie_get_pip / susie_get_posterior_mean / susie_get_posterior_sd / susie_get_posterior_samples / susie_get_cs` filter `V > prior_tol`. The class chain `c("mfsusie", "susie")` (model_class.R:148) ensures susieR-side dispatch on the model object falls through correctly. The ash and NIG branches in susieR are gated on `params$unmappable_effects %in% c("ash", "ash_filter_archived")` and `params$use_NIG` respectively; mfsusieR sets these to `"none"` and `FALSE` (mfsusie.R:374, 380) so neither branch fires. One coupling note: mfsusieR overrides BOTH `update_variance_components.mf_individual` (returns model directly, see line 476) AND `update_model_variance.mf_individual` (line 883-887); if only one were overridden, the susieR `.default` of the other would break because susieR expects `update_variance_components` to return `list(sigma2 = ...)` for `modifyList`-merge while mfsusieR's returns the model object. The pair is consistent but the two methods are tightly coupled.
- **Triage hint**: accept-pre-existing — pathway is clean.
