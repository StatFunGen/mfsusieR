## Findings

### BLOCKERS (resolved)

**B1. Task 6c.1 (degenerate-case test) ordered before task 7.1 (`mfsusie()`
implementation).** Task 6c.1 fits `mfsusie(...)` and compares to
`susieR::susie(...)`, but at the time 6c runs in the original order,
`mfsusie()` doesn't exist yet. Group 6c was placed between 6b and 7.

*Fix:* Moved former group 6c to a new group 7c, placed *after* group 7
(which implements `mfsusie()`). Tasks renumbered 6c.1 -> 7c.1, 6c.2 ->
7c.2. Group 7c carries an explicit "Depends on 7.1" header line.

**B2. CLAUDE.md hard rule #2 contradicts design.md D5.** Hard rule #2
said "Do not consult `fsusieR` for any design decision". Design.md D5
delegates per-scale wavelet-prior init, DWT helpers, smoothers, and
scale indexing to `fsusieR`. Same goes for review-loop methodology's
modularity audit, which expects fsusieR delegation as the *correct*
behavior.

*Fix:* Rewrote rule #2 to clarify: fsusieR is not a *paradigm source*
(do not model architecture decisions on it), but specific *runtime
utility delegations* (`fsusieR::remap_data`, `fsusieR::colScale`,
`fsusieR::gen_wavelet_indx`, `fsusieR::init_prior.default`,
smash/TI/HMM smoothers) ARE allowed and are documented in design.md
D5 / D8.

### NITs (resolved)

**N1. Stale path references in CLAUDE.md.** Phase 1 description and
Phase 5 description used bare `notes/paradigms/`, `notes/investigations/`
(without the `inst/` prefix). The actual layout puts these under
`inst/notes/`.

*Fix:* Replaced 5 path references in CLAUDE.md (Phase 1 file list
lines 67-89, Phase 5 lines 227-233). Session reporting at line 335
already used `inst/notes/sessions/`.

**N2. Hard rule #1 over-broad relative to the L_greedy override.** The
rule said susieR is read-only with no exception listed in CLAUDE.md
itself, but design.md authorizes a feature-branch addition.

*Fix:* Updated rule #1 to record the single authorized exception
(branch `feature/L_greedy` on `~/GIT/susieR`) with a pointer to
design.md's Migration Plan.

**N3. Hard rule #4 phase-ordering ambiguity.** The rule said "Do not
start phase N+1 while phase N is not archived in OpenSpec". But Phase
1 has no OpenSpec change (notes only), so the rule was technically
ambiguous about Phase 1 -> Phase 2.

*Fix:* Clarified rule #4 to note that Phase 1 -> Phase 2 only requires
Gao's review of paradigm notes; all other transitions require the
Phase N OpenSpec change to be archived.

### False positives (dismissed)

**FP1. "Test argument order is wrong" (Auditor 2).** Auditor 2 flagged
the Phase 4 example test in CLAUDE.md as a blocker because `mfsusie`
takes `(X, Y, ...)` but `multfsusie` takes `(Y, X, ...)`. Reading the
example carefully:

```r
fit_new <- mfsusie(data$X, data$Y, data$pos, L = 10, ...)
fit_old <- mvf.susie.alpha::multfsusie(data$Y, data$X, data$pos, L = 10)
```

These calls are CORRECT: each function gets called with its native
argument order, and the actual data values (`data$X` and `data$Y`)
are paired correctly. Verified by inspecting the original signature:
`multfsusie <- function(Y, X, L = 2, pos = NULL, ...)`. The auditor
appears to have read the call sites without checking the per-function
order. Dismissed.

### Non-blocking nits (deferred to Phase 3)

**NB1. Citation format mixed in design.md.** Some cites use
`eq:label`, others `line N`, others file paths. Will be unified to
`@manuscript_ref methods/<file>.tex eq:<label>` during Phase 3 roxygen
work per design D12.

**NB2. Some D-decisions (D2, D8b, D9, D10, D11, D12) not mirrored in
proposal.md "What Changes" bullets.** All are reflected in spec
requirements; proposal bullets cover the user-facing changes, not
every internal architectural decision. Acceptable.
